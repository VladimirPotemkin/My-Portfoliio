# План реализации SPA портфолио-резюме

## Контекст

Сейчас проект находится в стартовом состоянии: в репозитории есть базовый шаблон `Vue 3 + TypeScript + Vite`, без `Pinia`, `Vue Router`, `TailwindCSS`, `vue-i18n`, `@vueuse/core` и `@vueuse/motion`.

Цель следующего этапа: превратить шаблон в аккуратное одностраничное SPA-портфолио фронтенд-разработчика с маршрутизацией по вкладкам, переключением языка, светлой и тёмной темами, GitHub API, анимациями появления при скролле и модальным CV.

## Что считаем итоговым результатом

- SPA на `Vue 3` с `Composition API` и `script setup`.
- Везде строгий `TypeScript`, без `any`.
- Состояние вынесено в `Pinia`.
- Навигация построена через `Vue Router 4`.
- Стили на `TailwindCSS`.
- Локализация на `vue-i18n` с сохранением выбранного языка.
- Анимации появления через `@vueuse/motion`.
- Интеграция с GitHub API через стор.
- Светлая и тёмная темы с сохранением выбранного режима.
- Архитектура проекта соответствует структуре из ТЗ и остаётся удобной для расширения.

## Визуальное направление

На основе присланных референсов предлагаю собрать гибридную визуальную систему:

- Общий характер интерфейса: чистый, выразительный, с крупной типографикой и заметными секциями.
- Для разделов `Experience` и `Tech Stack`: ближе к последним рендерам с тёмным фоном, тонкими разделителями, акцентным бирюзово-мятным цветом и более "разработческим" настроением.
- Для карточек портфолио: взять от первых референсов аккуратную сетку, большие превью, воздух между блоками и выразительные заголовки.
- Для светлой темы: не инвертировать дизайн механически, а подготовить отдельные токены цветов, чтобы интерфейс оставался контрастным и собранным.

## Ключевые технические решения

### 1. Темизация

Вместо SCSS-миксинов лучше использовать дизайн-токены на CSS custom properties:

- `:root` для светлой темы.
- `.dark` на `html` для тёмной темы.
- Семантические переменные вроде `--color-bg`, `--color-surface`, `--color-text`, `--color-accent`.
- Tailwind будет использовать эти токены через кастомные utility-классы и/или расширение темы.

Почему так лучше:

- темы переключаются мгновенно;
- цвета можно менять централизованно;
- решение проще поддерживать вместе с Tailwind, чем ручные миксины.

### 2. Источник данных профиля

Контент портфолио лучше разделить на два типа:

- статические данные профиля, опыта, навыков, контактов и CV;
- динамические данные репозиториев из GitHub API.

Для этого стоит завести отдельный файл-конфиг с типизированными мок-данными профиля, чтобы потом легко заменить на реальные тексты.

### 3. GitHub API

Для репозиториев использовать стор `useGithubStore`:

- action `fetchRepos(username: string)`;
- фильтрация только репозиториев с описанием и без форков;
- вычисление списка языков для фильтра;
- состояния `idle/loading/success/error`;
- повторная загрузка по кнопке `Retry`.

Дополнительно имеет смысл предусмотреть:

- параметр `VITE_GITHUB_USERNAME` в `.env`;
- fallback на демонстрационный username, если переменная не задана;
- мягкую обработку rate limit ошибок GitHub API.

### 4. Локализация

Все строки выносятся в `src/locales/ru.json` и `src/locales/en.json`:

- меню;
- hero-блок;
- био;
- названия кнопок;
- сообщения ошибок;
- уведомления;
- подписи в модалке CV;
- подписи фильтров и состояний загрузки.

Выбранный язык сохраняется в `localStorage` и синхронизируется через стор.

### 5. Анимации

План по анимациям такой:

- переходы между маршрутами через `Transition` с мягким `fade`;
- появление блоков при скролле через `v-motion` и `visibleOnce`;
- модалка CV через `fade + scale`;
- карточки репозиториев и элементы таймлайна анимируются каскадно, но без перегруза.

## Что именно будет сделано в коде

На следующем шаге можно переходить к полной реализации:

- обновить `package.json`;
- настроить Tailwind и базовые стили;
- переписать `main.ts` и `App.vue`;
- создать router, stores, types, locales, components, views;
- подключить motion, i18n и theme management;
- реализовать GitHub API;
- собрать весь UI по новым референсам.

## Допущения по умолчанию, если не ждать ответов

- Использовать демонстрационный `GitHub username`: `torvalds`.
- Контент CV и bio временно заполнить качественными мок-данными фронтенд-разработчика.
- Контактную форму оставить опциональной и пока не подключать к backend.
- Иконки брать из inline SVG, чтобы не добавлять лишнюю библиотеку.
- Для роутинга использовать `createWebHistory`, если не планируется деплой на GitHub Pages.
- Основной визуальный режим проектировать вокруг тёмной темы, но сразу поддержать и светлую.

## Риски и нюансы

- У GitHub API есть rate limit без токена. Для демо это терпимо, но в проде может понадобиться proxy или token.
- Если планируется GitHub Pages, нужно заранее выбрать `createWebHashHistory` или настроить fallback.
- Без реального контента био и CV интерфейс будет качественным, но не персонализированным.
- Слишком много анимации может перегрузить страницу, поэтому motion стоит держать умеренным.
